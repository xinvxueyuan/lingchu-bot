# Design Prototyping

Use this skill to make fuzzy design space concrete before committing to implementation.

## Route

- For multiple API/module/interface shapes, read `references/design-an-interface.md`.
- For stress-testing a plan, terminology, architecture, or tradeoff set through questions, read `references/design-grill.md`.
- For a runnable throwaway prototype or UI variations, read `references/prototype.md`.

## Working Rules

- Start with the decision the prototype or interview must clarify.
- Keep prototype artifacts disposable unless the user asks to productionize them.
- When building UI prototypes, make the first screen useful rather than a landing page.
- End with the design choice, the evidence from the prototype or interview, and remaining risks.
- Use `gitnexus-exploring` to understand existing architecture before designing new interfaces.
- Use `gitnexus-impact-analysis` to assess the impact of proposed design changes.
