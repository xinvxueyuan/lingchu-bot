## Troubleshooting

This page collects the most common local startup and group management issues currently covered by the project.

## Installation and startup

<Accordions type="single">
  <Accordion title="Dependency installation fails">

First confirm that Python and uv are available:

```bash
python --version
uv --version
```

The project requires Python 3.13. Dependency installation should use:

```bash
uv sync --frozen
```

  </Accordion>
  <Accordion title="Boolean configuration errors">

If you see an `in_containers 配置错误` error, check `.env` or the NoneBot configuration file.

Correct:

```dotenv
LINGCHU_IN_CONTAINERS=true
```

Incorrect:

```dotenv
LINGCHU_IN_CONTAINERS=True
```

NoneBot configuration files only accept standard JSON lowercase `true` / `false`.

  </Accordion>
  <Accordion title="Bot cannot connect to the platform">

Whether the service for the enabled adapter has started; the default is OneBot V11. Whether NoneBot and the enabled adapter connection settings match. Whether the account, network, firewall, and platform permissions are usable.

  </Accordion>
</Accordions>

## Adapter selection

<Accordions type="single">
  <Accordion title="LINGCHUAdapter is unset">

When `LINGCHUAdapter` is unset, Lingchu Bot enables `~onebot.v11` by default; make sure NoneBot has loaded OneBot V11.

  </Accordion>
  <Accordion title="PlatformAdapterNotLoadedError">

The selected adapter was not loaded or registered by NoneBot. Load the matching adapter or change `LINGCHUAdapter` to one that is already loaded.

  </Accordion>
  <Accordion title="Multiple adapters for the same platform">

Do not explicitly configure multiple adapters for the same platform. For example, `LINGCHUAdapter = "~onebot.v11+~another_adapter"` fails immediately.

  </Accordion>
  <Accordion title="Unknown adapter registered at runtime">

Extra registered same-platform adapters do not create a conflict. Lingchu Bot treats unselected adapters as disabled. Unknown adapters that do not belong to any platform profile are not considered enabled; message storage writes their `platform` field as `unknown`.

  </Accordion>
</Accordions>

## Group management

<Accordions type="single">
  <Accordion title="Mute or unmute command fails">

Mute, unmute, and whole-group mute depend on platform permissions. When they fail, check whether the bot is in the target group, whether the bot has the required management permission, whether the target user can be operated on by the bot, and whether the enabled adapter API reports a network error or rejected operation.

  </Accordion>
  <Accordion title="Feedback appears in the wrong language">

Check whether `LINGCHU_LOCALE` is set to a supported catalog such as `zh_CN` or `en_US`. Unknown text falls back to the original message.

  </Accordion>
</Accordions>

## CI or local checks

<Accordions type="single">
  <Accordion title="Check commands fail">

Read the failing command and line number, then fix only the related scope. Common checks include:

```bash
uv run -m ruff check . --output-format=github
uv run -m ruff format --check .
uv run -m pyright
uv run -m ty check --output-format github
uv run -m pytest
```

  </Accordion>
</Accordions>

</Callout>
