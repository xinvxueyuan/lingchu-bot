## Troubleshooting

This page collects the most common local startup and group management issues currently covered by the project.

## Installation and startup

<Accordions type="single">
  <Accordion title="Dependency installation fails">

First confirm that Python and uv are available:

```bash
python --version
uv --version
```

The project requires Python 3.13. Dependency installation should use:

```bash
uv sync --frozen
```

  </Accordion>
  <Accordion title="Boolean configuration errors">

If you see an `in_containers 配置错误` error, check `.env` or the NoneBot configuration file.

Correct:

```dotenv
LINGCHU_IN_CONTAINERS=true
```

Incorrect:

```dotenv
LINGCHU_IN_CONTAINERS=True
```

NoneBot configuration files only accept standard JSON lowercase `true` / `false`.

  </Accordion>
  <Accordion title="Bot cannot connect to the platform">

Whether the service for the enabled adapter has started; the default is OneBot V11. Whether NoneBot and the enabled adapter connection settings match. Whether the account, network, firewall, and platform permissions are usable.

  </Accordion>
</Accordions>

## Adapter selection

<Accordions type="single">
  <Accordion title="LINGCHUAdapter is unset">

When `LINGCHUAdapter` is unset, Lingchu Bot enables `~onebot.v11` by default; make sure NoneBot has loaded OneBot V11.

  </Accordion>
  <Accordion title="PlatformAdapterNotLoadedError">

The selected adapter was not loaded or registered by NoneBot. Load the matching adapter or change `LINGCHUAdapter` to one that is already loaded.

  </Accordion>
  <Accordion title="Multiple adapters for the same platform">

Do not explicitly configure multiple adapters for the same platform. For example, `LINGCHUAdapter = "~onebot.v11+~another_adapter"` fails immediately.

  </Accordion>
  <Accordion title="Unknown adapter registered at runtime">

Extra registered same-platform adapters do not create a conflict. Lingchu Bot treats unselected adapters as disabled. Unknown adapters that do not belong to any platform profile are not considered enabled; message storage writes their `platform` field as `unknown`.

  </Accordion>
</Accordions>

## Group management

<Accordions type="single">
  <Accordion title="Mute or unmute command fails">

Mute, unmute, and whole-group mute depend on platform permissions. When they fail, check whether the bot is in the target group, whether the bot has the required management permission, whether the target user can be operated on by the bot, and whether the enabled adapter API reports a network error or rejected operation.

  </Accordion>
  <Accordion title="Feedback appears in the wrong language">

Check whether `LINGCHU_LOCALE` is set to a supported catalog such as `zh_CN` or `en_US`. Unknown text falls back to the original message.

  </Accordion>
  <Accordion title="Announcement image is not sent">

OneBot V11 group announcements use implementation-specific APIs. For NapCat, `_send_group_notice` expects the `image` value to be a filesystem path readable by the NapCat side. Common causes:

- Lingchu Bot wrote the cached image on Windows, but the NapCat Docker container cannot read that path.
- `LINGCHU_ANNOUNCEMENT_IMAGE_CACHE_DIR` and `LINGCHU_ANNOUNCEMENT_IMAGE_PROTOCOL_DIR` are unset or point to different directories.
- The Docker bind mount was changed without recreating the NapCat container.
- Lingchu Bot was not restarted after changing `.env`.
- NapCat is below the supported version or the current implementation is not detected as `NapCat.Onebot`.

For Windows plus Docker Desktop, configure the path bridge described in [NapCat Docker](../platforms/qq/frameworks/napcat-docker), then verify that a file written under `LINGCHU_ANNOUNCEMENT_IMAGE_CACHE_DIR` is visible inside the container at `LINGCHU_ANNOUNCEMENT_IMAGE_PROTOCOL_DIR`.

Additional checks for WSL2 (Debian / Ubuntu) deployments:

- **Docker Desktop File Sharing allow-list** (this is the most common WSL2 + Docker Desktop pitfall and must be done first). Open Docker Desktop → Settings → Resources → File sharing and confirm `\\wsl.localhost\Debian\` is in the list (replace `Debian` with your distro name; older WSL uses `\\wsl$\Debian\`, the trailing `\` is required). When this is missing, the container's `/lingchu-bot` will be an empty directory; `docker inspect` will not flag the problem, but `docker exec napcat mount | grep lingchu` shows `type overlay` (correct state is `fuse.bind`). After editing, click **Apply & restart** so Docker Desktop rebuilds the WSL integration index.
- Is `LINGCHU_ANNOUNCEMENT_IMAGE_CACHE_DIR` still a Windows-style drive-letter path (e.g. `C:/dev/lingchu-bot/...`)? On WSL2 `pathlib` treats it as a relative path, the startup log emits a `Announcement image cache dir ... is incompatible with the current system Linux` WARNING, and NapCat returns `retcode=1200, image field format may be incorrect` when sending the announcement. Change it to a POSIX path inside WSL2 (e.g. `/home/xinvdev/lingchu-bot/.local/napcat-announcement-images`) and recreate the NapCat container.
- Is the WSL2 integration enabled in Docker Desktop → Settings → Resources → WSL Integration? (default is on, but some distros must be toggled individually).
- After recreating the container, run `docker exec napcat ls -la /lingchu-bot/.local/napcat-announcement-images/` to verify the bind mount is actually live. `docker inspect` alone is not enough — when File Sharing is missing the WSL2 integration layer returns an empty directory without raising an error.

For complete WSL2 deployment examples (Docker Desktop + WSL integration as the recommended default, UNC path as the locked-down fallback, and a native dockerd-in-WSL2 alternative), see the [NapCat Docker WSL2 deployment section](../platforms/qq/frameworks/napcat-docker#wsl2-deployment).

  </Accordion>
</Accordions>

## CI or local checks

<Accordions type="single">
  <Accordion title="Check commands fail">

Read the failing command and line number, then fix only the related scope. Common checks include:

```bash
uv run -m ruff check . --output-format=github
uv run -m ruff format --check .
uv run -m pyright
uv run -m ty check --output-format github
uv run -m pytest
```

  </Accordion>
</Accordions>

If your issue is not listed here, check the [Configuration](configuration) page for detailed settings, or open an issue at [GitHub](https://github.com/xinvxueyuan/lingchu-bot/issues).

</Callout>
