## Configuration

Lingchu Bot stores lightweight runtime settings in a plugin-owned JSON5 configuration file while reusing NoneBot2's global configuration parser.

On first startup, the plugin creates `config.json5`. Its path is provided by `nonebot-plugin-localstore` and belongs to the plugin configuration directory, not the data or cache directory.

Runtime configuration priority, from highest to lowest, is:

1. OS environment variables
2. NoneBot dotenv / global configuration
3. `config.json5`
4. Code defaults

OS environment variable and dotenv loading, case handling, Boolean parsing, path parsing, and env-file selection follow NoneBot's own behavior. Lingchu Bot does not implement separate dotenv discovery.

## Runtime settings

| Setting                                                  | Default           | Description                                                             |
| -------------------------------------------------------- | ----------------- | ----------------------------------------------------------------------- |
| `superuser_key`                                          | `123456789abcdef` | Superuser authentication key                                            |
| `message_store_enabled`                                  | `true`            | Whether to enable message-store runtime hooks                           |
| `message_store_retention_days`                           | `30`              | Number of days to retain message records; `0` disables day-based expiry |
| `message_store_summary_limit`                            | `500`             | Maximum length for text, data, and result summaries                     |
| `message_store_record_api_calls`                         | `true`            | Whether to record platform API call summaries                           |
| `message_store_cleanup_enabled`                          | `true`            | Whether to clean expired message records during shutdown                |
| `lingchu_adapter` / `LINGCHUAdapter` / `LINGCHU_ADAPTER` | unset             | Selects the enabled adapter for each platform                           |

Example `config.json5`:

```json5
{
  superuser_key: "123456789abcdef",
  message_store_enabled: true,
  message_store_retention_days: 30,
  message_store_summary_limit: 500,
  message_store_record_api_calls: true,
  message_store_cleanup_enabled: true,
  lingchu_adapter: "~onebot.v11",
}
```

NoneBot global configuration can still override JSON5:

```toml
LINGCHUAdapter = "~onebot.v11"
```

Or in a `.env` file:

```dotenv
LINGCHUAdapter=~onebot.v11
```

If the same key appears in OS environment variables, dotenv, and `config.json5`, the OS environment variable wins.

## Core path settings

`core_version`, `data_dir`, `config_dir`, `cache_dir`, and system platform helpers are still provided by the core `Config`. Paths come from `nonebot-plugin-localstore`:

- `data_dir`: data file directory.
- `config_dir`: configuration file directory.
- `cache_dir`: cache file directory.

## Adapter selection

Lingchu Bot groups concrete adapters into platform capabilities. Only one adapter is enabled per platform by default. The current QQ platform priority is:

1. `~onebot.v11`
2. `~milky`
3. `~qq`
4. `~onebot.v12`

When unset, QQ uses `~onebot.v11`. To select an adapter explicitly, set a global NoneBot configuration value:

```toml
LINGCHUAdapter = "~onebot.v11"
```

Only the QQ platform profile is implemented at the moment, so QQ selects `~onebot.v11` in this example. To enable Milky:

```toml
LINGCHUAdapter = "~milky"
```

Explicitly configuring multiple known adapters for the same platform fails startup:

```toml
LINGCHUAdapter = "~milky+~onebot.v11"
```

Lingchu Bot does not control which adapters NoneBot actually imports or registers; it only selects business code through `LINGCHUAdapter` and checks that the selected adapter has been loaded and registered by NoneBot. When unset, Lingchu selects `~onebot.v11` by default, so OneBot V11 must be loaded. If Milky is selected explicitly but NoneBot has not loaded Milky, startup fails as well. Other same-platform adapters may still be imported or registered, but Lingchu Bot treats them as disabled. They do not participate in platform resolution, message storage, or API call recording. Explicitly declaring an adapter that Lingchu has not implemented or cannot recognize in `LINGCHUAdapter` fails startup; extra unknown adapters registered at runtime are still ignored.

An unknown adapter that does not belong to any platform profile is not considered enabled. Message storage still accepts its events, but writes the `platform` field as `unknown`. This gives downstream displays a stable Unknown bucket instead of treating the raw adapter name as a platform ID.

## Internationalization settings

Runtime translation reads these NoneBot configuration keys in order:

1. `lingchu_locale`
2. `lc_locale`
3. `locale`

The recommended project-specific key in `.env` is:

```dotenv
LINGCHU_LOCALE=zh_CN
```

Available catalogs currently include `zh_CN` and `en_US`. Locale names are normalized before use, so `en-US` and `en_US.UTF-8` both become `en_US`. When the setting is missing, empty, or NoneBot has not been initialized, the default locale is `zh_CN`.

## Container environment flag

`in_containers` comes from global NoneBot configuration. It must be a Boolean value.

```dotenv
IN_CONTAINERS=true
```

Do not write:

```dotenv
IN_CONTAINERS=True
```

NoneBot configuration files only accept standard JSON lowercase `true` / `false`. If a string-like Boolean is passed in, the project raises a clear configuration error.

## Local runtime paths

The repository root currently does not ship a committed local `bot.py`. Project-level localstore paths are controlled by NoneBot configuration; see `.env.example`:

```dotenv
LOCALSTORE_USE_CWD=true
```

This means localstore-related directories prefer the current working directory, which is convenient for local development and debugging.
