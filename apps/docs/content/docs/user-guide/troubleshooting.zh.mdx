## 故障排除

本页收集项目当前覆盖的最常见的本地启动和群管理问题。

## 安装与启动

<Accordions type="single">
  <Accordion title="依赖安装失败">

首先确认 Python 和 uv 是否可用：

```bash
python --version
uv --version
```

项目要求 Python 3.13。依赖安装应使用：

```bash
uv sync --frozen
```

  </Accordion>
  <Accordion title="布尔配置错误">

如果看到 `in_containers 配置错误`，请检查 `.env` 或 NoneBot 配置文件。

正确写法：

```dotenv
LINGCHU_IN_CONTAINERS=true
```

错误写法：

```dotenv
LINGCHU_IN_CONTAINERS=True
```

NoneBot 配置文件只接受标准 JSON 小写 `true` / `false`。

  </Accordion>
  <Accordion title="Bot 无法连接平台">

请确认已启用适配器的服务是否已启动（默认为 OneBot V11），NoneBot 与已启用适配器的连接设置是否匹配，以及账号、网络、防火墙和平台权限是否可用。

  </Accordion>
</Accordions>

## 适配器选择

<Accordions type="single">
  <Accordion title="LINGCHUAdapter 未设置">

当 `LINGCHUAdapter` 未设置时，Lingchu Bot 默认启用 `~onebot.v11`；请确保 NoneBot 已加载 OneBot V11。

  </Accordion>
  <Accordion title="PlatformAdapterNotLoadedError">

选中的适配器未被 NoneBot 加载或注册。请加载对应的适配器，或将 `LINGCHUAdapter` 改为已加载的适配器。

  </Accordion>
  <Accordion title="同一平台配置了多个适配器">

不要为同一平台显式配置多个适配器。例如 `LINGCHUAdapter = "~onebot.v11+~another_adapter"` 会立即失败。

  </Accordion>
  <Accordion title="运行时注册了未知适配器">

额外注册的同平台适配器不会产生冲突。Lingchu Bot 将未选中的适配器视为已禁用。不属于任何平台 profile 的未知适配器不被视为已启用；消息存储会将其 `platform` 字段写入 `unknown`。

  </Accordion>
</Accordions>

## 群管理

<Accordions type="single">
  <Accordion title="禁言或解禁命令失败">

禁言、解禁和全体禁言依赖平台权限。失败时请检查 Bot 是否在目标群中、是否拥有所需管理权限、目标用户是否可被 Bot 操作，以及已启用适配器 API 是否报告网络错误或操作被拒绝。

  </Accordion>
  <Accordion title="反馈语言不正确">

检查 `LINGCHU_LOCALE` 是否设置为支持的目录，如 `zh_CN` 或 `en_US`。未知文本会回退到原始消息。

  </Accordion>
  <Accordion title="群公告图片发不出去">

OneBot V11 群公告使用实现专属 API。对 NapCat 来说，`_send_group_notice` 的 `image` 值必须是 NapCat 侧可读取的文件系统路径。常见原因：

- Lingchu Bot 在 Windows 写入了缓存图片，但 NapCat Docker 容器无法读取这个路径。
- `LINGCHU_ANNOUNCEMENT_IMAGE_CACHE_DIR` 与 `LINGCHU_ANNOUNCEMENT_IMAGE_PROTOCOL_DIR` 未设置，或没有指向同一个挂载目录。
- 修改了 Docker bind mount，但没有重建 NapCat 容器。
- 修改 `.env` 后没有重启 Lingchu Bot。
- NapCat 版本低于支持范围，或当前实现未被识别为 `NapCat.Onebot`。

Windows 加 Docker Desktop 场景请按[NapCat Docker](../platforms/qq/frameworks/napcat-docker)配置路径穿透，并验证写入 `LINGCHU_ANNOUNCEMENT_IMAGE_CACHE_DIR` 的文件能在容器内通过 `LINGCHU_ANNOUNCEMENT_IMAGE_PROTOCOL_DIR` 读取。

WSL2（Debian / Ubuntu）部署的额外检查项：

- **Docker Desktop File Sharing 白名单**（这是 WSL2 + Docker Desktop 集成最常见的坑，必须**先**做这一步）。打开 Docker Desktop → Settings → Resources → File sharing，确认 `\\wsl.localhost\Debian\` 已加入白名单（distro 名要换成你自己的，旧版 WSL 写 `\\wsl$\Debian\`，**`\` 结尾不能漏**）。漏配时容器内 `/lingchu-bot` 会是个空目录，`docker inspect` 看不出问题，但 `docker exec napcat mount | grep lingchu` 会显示 `type overlay`（正常应是 `fuse.bind`）。修改后必须点 **Apply & restart** 让 Docker Desktop 重建 WSL 集成索引。
- `LINGCHU_ANNOUNCEMENT_IMAGE_CACHE_DIR` 是否仍是 Windows 风格盘符路径（如 `C:/dev/lingchu-bot/...`）。在 WSL2 上 `pathlib` 会把它当相对路径，启动日志会输出 `公告图片缓存目录 ... 与当前系统 Linux 不兼容` 的 WARNING，发送公告图片时 NapCat 会返回 `retcode=1200, image 字段可能格式不正确`。请改成 WSL2 内的 POSIX 路径（例如 `/home/xinvdev/lingchu-bot/.local/napcat-announcement-images`），并重建 NapCat 容器。
- Docker Desktop 与 WSL2 集成是否在 Docker Desktop → Settings → Resources → WSL Integration 中开启（默认开启，但有的发行版需要单独勾选）。
- 重建容器后必须做一次 `docker exec napcat ls -la /lingchu-bot/.local/napcat-announcement-images/` 验证 bind mount 真的有效；只看 `docker inspect` 不够，WSL2 集成层可能在 File Sharing 缺失时返回一个空目录而 `inspect` 不报错。

完整 WSL2 部署示例（Docker Desktop + WSL 集成、UNC 路径备选、WSL2 原生 dockerd 三种拓扑）见 [NapCat Docker WSL2 部署章节](../platforms/qq/frameworks/napcat-docker#wsl2-部署)。

  </Accordion>
</Accordions>

## CI 或本地检查

<Accordions type="single">
  <Accordion title="检查命令失败">

阅读失败的命令和行号，然后只修复相关范围。常用检查包括：

```bash
uv run -m ruff check . --output-format=github
uv run -m ruff format --check .
uv run -m pyright .
uv run -m ty check --output-format github
uv run -m pytest
```

  </Accordion>
</Accordions>
