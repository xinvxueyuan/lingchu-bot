title: Adapter Selection
---

Lingchu Bot groups concrete adapters into platform capabilities. Only one adapter is enabled per platform by default. The current QQ platform priority is:

1. `~onebot.v11`

When unset, QQ uses `~onebot.v11`. To select an adapter explicitly, set a global NoneBot configuration value:

```toml
LINGCHUAdapter = "~onebot.v11"
```

Default selection. Make sure NoneBot has loaded the OneBot V11 adapter. Other adapter IDs (such as Milky, QQ, or OneBot V12) are not implemented; configuring them raises `PlatformAdapterUnknownError`.

### Environment Variable Form

Lingchu Bot also reads the `LINGCHU_ADAPTER` environment variable, which is convenient for containerized deployments and CI where editing `.env` or TOML is impractical. The underlying `lingchu_adapter` field accepts a single adapter string, a `+`-separated string of multiple IDs, or a list of strings:

```bash
# Single adapter
LINGCHU_ADAPTER=~onebot.v11

# Multiple adapters in string form
LINGCHU_ADAPTER=~onebot.v11+~another_adapter
```

When set via NoneBot's dotenv/TOML config, a JSON array literal is also accepted:

```toml
LINGCHUAdapter = ["~onebot.v11", "~another_adapter"]
```

Configuration priority (highest first):

1. OS environment variables (`LINGCHU_ADAPTER` in `os.environ`)
2. NoneBot dotenv file (`.env`, where `LINGCHUAdapter` or `LINGCHU_ADAPTER` may appear)
3. NoneBot `global_config` (resolved through the `lingchu_adapter` alias)

<Callout type="warn" title="Conflict prevention">

Explicitly configuring multiple known adapters for the same platform fails startup:

```toml
LINGCHUAdapter = "~onebot.v11+~another_adapter"
```

Lingchu Bot does not control which adapters NoneBot actually imports or registers; it only selects business code through `LINGCHUAdapter` and checks that the selected adapter has been loaded and registered by NoneBot.

</Callout>

An unknown adapter that does not belong to any platform profile is not considered enabled. Message storage still accepts its events, but writes the `platform` field as `unknown`. This gives downstream displays a stable Unknown bucket instead of treating the raw adapter name as a platform ID.

For developer documentation on the adapter registry and adding new platforms, see [Adapter Guide](/docs/developer-guide/architecture/adapter-guide).
