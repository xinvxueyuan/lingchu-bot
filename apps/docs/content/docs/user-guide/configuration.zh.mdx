## 配置

Lingchu Bot 将轻量级运行时设置存储在插件专属的 JSON5 配置文件中，同时复用 NoneBot2 的全局配置解析器。

首次启动时，插件会创建 `config.json5`。其路径由 `nonebot-plugin-localstore` 提供，属于插件配置目录，而非数据或缓存目录。

<Callout title="配置优先级">

运行时配置优先级从高到低：

1. 操作系统环境变量
2. NoneBot dotenv / 全局配置
3. `config.json5`
4. 代码默认值

操作系统环境变量和 dotenv 加载、大小写处理、布尔值解析、路径解析和 env 文件选择遵循 NoneBot 自身的行为。Lingchu Bot 不实现单独的 dotenv 发现。

</Callout>

`config.json5` 示例：

```json5
{
  superuser_key: "123456789abcdef",
  message_store_enabled: true,
  message_store_retention_days: 30,
  message_store_summary_limit: 500,
  message_store_record_api_calls: true,
  message_store_cleanup_enabled: true,
  lingchu_adapter: "~onebot.v11",
  lingchu_superusers: null,
}
```

NoneBot 全局配置仍可覆盖 JSON5：

```toml
LINGCHUAdapter = "~onebot.v11"
```

或在 `.env` 文件中：

```dotenv
LINGCHUAdapter=~onebot.v11
```

如果同一键同时出现在操作系统环境变量、dotenv 和 `config.json5` 中，操作系统环境变量优先。

## 超级用户身份设置

Lingchu 权限系统使用稳定 UID 和平台账号绑定，而不是只把平台账号 ID 当作身份。推荐用 JSON5 配置专属超级用户映射：

```dotenv
LINGCHU_SUPERUSERS='{"userA":{"qq":123,"telegram":"tg-id","discord":"discord-id"}}'
```

对象键是 Lingchu UID；每个值将 `qq`、`telegram`、`discord` 等平台 ID 映射到该用户在对应平台的账号 ID。

如果未设置 `LINGCHU_SUPERUSERS`，Lingchu 会兼容导入 NoneBot 原生 `SUPERUSERS`，将其作为 QQ 平台账号并按顺序生成 `user1` 到 `user999`。当两个来源都不存在、值不是 UID 到平台账号的映射，或同一平台账号绑定到多个 UID 时，启动会失败。

SUPERUSERS 始终通过 Lingchu 命令权限检查，但平台 API 限制和命令自身的业务防呆仍然生效。

## 核心路径设置

`core_version`、`data_dir`、`config_dir`、`cache_dir` 和系统平台辅助信息仍由核心 `Config` 提供。路径来自 `nonebot-plugin-localstore`：

| 路径         | 用途         |
| ------------ | ------------ |
| `data_dir`   | 数据文件目录 |
| `config_dir` | 配置文件目录 |
| `cache_dir`  | 缓存文件目录 |

### 群公告图片路径桥接

OneBot V11 群公告使用实现专属的 `_send_group_notice` API。NapCat 当前会把公告 `image` 字段作为协议端本机文件路径读取。当 Lingchu Bot 运行在 Windows，而 NapCat 运行在 Docker Desktop 容器中时，需要配置共享目录，并向 NapCat 传入容器内路径：

```dotenv
ANNOUNCEMENT_IMAGE_CACHE_DIR=C:/dev/lingchu-bot/.local/napcat-announcement-images
ANNOUNCEMENT_IMAGE_PROTOCOL_DIR=/lingchu/announcement-images
```

将同一个宿主机目录挂载到 NapCat 容器：

```yaml
volumes:
  - C:/dev/lingchu-bot/.local/napcat-announcement-images:/lingchu/announcement-images:ro
```

未设置这些配置时，公告图片仍写入普通的 `cache_dir / "announcement_images"` 路径，保持既有的同文件系统行为。

## 适配器选择

Lingchu Bot 将具体适配器按平台能力分组。每个平台默认只启用一个适配器。当前 QQ 平台优先级为：

1. `~onebot.v11`
2. `~milky`
3. `~qq`
4. `~onebot.v12`

未设置时，QQ 使用 `~onebot.v11`。要显式选择适配器，设置 NoneBot 全局配置值：

<Tabs items={['OneBot V11', 'Milky']}>
  <Tab value="OneBot V11">

```toml
LINGCHUAdapter = "~onebot.v11"
```

默认选择。请确保 NoneBot 已加载 OneBot V11 适配器。

  </Tab>
  <Tab value="Milky">

```toml
LINGCHUAdapter = "~milky"
```

需要 NoneBot 已加载 Milky 适配器。

  </Tab>
</Tabs>

<Callout type="warn" title="冲突预防">

为同一平台显式配置多个已知适配器会导致启动失败：

```toml
LINGCHUAdapter = "~milky+~onebot.v11"
```

Lingchu Bot 不控制 NoneBot 实际导入或注册哪些适配器；它只通过 `LINGCHUAdapter` 选择业务代码，并校验选中的适配器已被 NoneBot 加载和注册。

</Callout>

不属于任何平台 profile 的未知适配器不被视为已启用。消息存储仍会接受其事件，但会将 `platform` 字段写入 `unknown`。这为下游展示提供了稳定的 Unknown 分桶，而非将原始适配器名称视为平台 ID。

## 国际化设置

运行时翻译按以下顺序读取 NoneBot 配置键：

1. `lingchu_locale`
2. `lc_locale`
3. `locale`

推荐的 `.env` 项目专用键：

```dotenv
LINGCHU_LOCALE=zh_CN
```

当前可用目录包括 `zh_CN` 和 `en_US`。locale 名称在使用前会进行规范化，因此 `en-US` 和 `en_US.UTF-8` 都会变为 `en_US`。当设置缺失、为空或 NoneBot 尚未初始化时，默认 locale 为 `zh_CN`。

## 容器环境标志

`in_containers` 来自 NoneBot 全局配置。必须为布尔值。

```dotenv
IN_CONTAINERS=true
```

<Callout type="warn" title="布尔值格式">

不要写 `IN_CONTAINERS=True`。NoneBot 配置文件只接受标准 JSON 小写 `true` / `false`。如果传入字符串形式的布尔值，项目会抛出明确的配置错误。

</Callout>

## 本地运行路径

仓库根目录当前不包含已提交的本地 `bot.py`。项目级 localstore 路径由 NoneBot 配置控制；参见 `.env.example`：

```dotenv
LOCALSTORE_USE_CWD=true
```

这意味着 localstore 相关目录优先使用当前工作目录，便于本地开发和调试。
