## 开发流程

开发时保持改动范围小、验证明确、影响面可解释。仓库使用 `Taskfile.yml` 作为自动化主入口：Python/code 侧通过 `uv`，docs/frontend 侧通过 `pnpm`，跨工作区任务通过 Turbo 编排。

## 推荐步骤

1. 明确目标、成功标准和不做的范围。
2. 运行 `git status --short`，确认已有改动。
3. 阅读相关代码和测试，优先沿用现有结构。
4. 修改函数、类或方法前运行 GitNexus/codegraph 影响分析。
5. 实施最小改动并补充测试。
6. 通过 `task check`、`task test` 或更聚焦的 `uv` / `pnpm` 命令运行对应检查。
7. 提交前运行 GitNexus detect changes。

修改可翻译字符串时，还需要同步 Babel catalog。流程见 [国际化](i18n)。

## 自动化入口

优先使用这些 Taskfile 任务：

```bash
task install
task check
task test
task build
task ci
task i18n
```

只改 Python 代码时可以使用 `uv run ...` 聚焦验证；只改 docs 或前端包时可以使用 `pnpm --filter docs ...` 或 `pnpm turbo run ...` 聚焦验证。

## 辅助工具映射

- GitNexus / codegraph：理解代码结构、影响面和变更范围。
- Context7：查询库、框架、SDK、CLI 或云服务的当前文档。
- Husky / prek：本地提交钩子和 pre-commit 检查。
- Turbo：docs 与 packages 的跨工作区 lint、type check 和 build 编排。

## 保护工作区

不要回退、格式化或重写与当前任务无关的文件。遇到已有改动时，默认认为它们来自其他贡献者或前一轮工作。

## 分支和提交

常见分支名前缀：

- `feature/`
- `fix/`
- `hotfix/`
- `releases/`
- `docs/`
- `refactor/`

GitHub Actions 还会匹配 `feature/**`、`fix/**`、`hotfix/*`、`releases/**`、`main` 和 `dev` 等分支。具体目标分支以维护者或 PR 页面要求为准。

提交信息必须遵循 gitmoji + Conventional Commits。完整说明见 [提交规范](commit-style)。

```text
🐛 fix: 修复成员禁言成功反馈
📝 docs: 重写文档站
✅ test: 增加群管理异常覆盖
```

## PR 描述

PR 应说明：

- 改动目的。
- 关键实现。
- GitNexus 影响分析结果。
- 已运行的检查命令。
- 未完成事项或已知风险。

如果 PR 覆盖文档站或国际化，请说明是否已经运行 Markdown lint、docs lint/test/build 和 `task i18n`。
