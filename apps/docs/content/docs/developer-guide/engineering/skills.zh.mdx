## 代理技能

项目内置一组**技能（Skills）**——由 AI 编码代理（Trae、Codex、Claude Code）按需加载的结构化提示。技能将常见工作流转化为可重复、有纪律的流程。

技能**不会**注入每次对话。仅在用户任务匹配技能触发条件时才加载，保持上下文精简的同时让专业工作流随时可用。

## 开发调度链

计划到提交的 skills 组成五阶段流水线。在每个阶段开始时加载对应 skill；不要预加载整条链。

```text
grill-with-docs          ← 阶段 1：PLAN
  ↓                        grill 计划，构建 CONTEXT.md + ADRs
domain-modeling          ← 阶段 1b：磨砺领域语言（可选）
  ↓
to-spec                  ← 阶段 2：SPEC
  ↓                        将计划综合成 spec
to-tickets               ← 阶段 3：TICKETS
  ↓                        将 spec 拆成垂直切片 tickets
tdd                      ← 阶段 4：IMPLEMENT
  ↓                        红-绿-重构，一次一个切片
  ├─ ponytail             ← 实现期间强制最小方案
  ├─ context7-cli         ← 需要时查库文档
  ├─ gitnexus             ← 编辑任何 symbol 前运行 impact()
  ├─ firecrawl-*          ← 需要时进行 web 调研/抓取
  └─ react-doctor         ← 前端变更的 React triage
prek                     ← 阶段 5：COMMIT
                           Git 钩子：lint + format + type + test
```

轻量替代：当只需压力测试、不需要文档产物时，用 `grilling` / `grill-me` 替代 `grill-with-docs` + `domain-modeling`。

## 技能管理

技能由 [`skills` CLI](https://github.com/vercel-labs/skills) 管理，并在项目级追踪，确保每位贡献者拿到同一套技能。

| 关注点 | 位置 |
| --- | --- |
| 真实技能文件（单一来源） | `.agents/skills/<skill-name>/SKILL.md` |
| Claude Code 入口（整目录软链） | `.claude/skills` → `../.agents/skills` |
| Trae 入口（整目录软链） | `.trae/skills` → `../.agents/skills` |
| 锁文件（用 `skills experimental_install` 恢复） | `skills-lock.json` |
| GitNexus 管理的技能（gitignored，由 CLI 重建） | `.agents/skills/gitnexus/` |

新增技能：`npx -y skills add <owner/repo> --full-depth --skill '*' --agent codex,claude-code,trae-cn --yes`（不要加 `-g`，让它落到项目而非用户家目录）。新克隆后恢复：`npx -y skills experimental_install`。

<Callout title="避免使用 --all">

`--all` 是 `--skill '*' --agent '*' -y` 的简写，会向 CLI 识别的**所有** agent（约 30+）安装，包括 `eve`（`agent/skills/`）和 `astrbot`（`data/skills/`），它们会创建不需要的真实副本目录。始终用 `--agent codex,claude-code,trae-cn` 限定目标，只安装到本项目使用的 agent。

</Callout>

## 项目本地技能

### `prek` — Git 钩子

配置、运行或排查 `prek`、Husky、pre-commit、commit-msg 以及 lint/format/type/test 钩子。来源：[`j178/prek`](https://github.com/j178/prek)。

### GitNexus — 代码智能

GitNexus 从仓库构建知识图谱，并通过 MCP 工具提供影响分析、执行流探索、安全重命名和 PR 审查。技能目录由 `gitnexus analyze` 重建，被 gitignored。完整指南见 [GitNexus 工作流](gitnexus)。

| 技能 | 用途 |
| --- | --- |
| `gitnexus-exploring` | 理解架构和执行流 |
| `gitnexus-impact-analysis` | 修改前评估爆炸半径 |
| `gitnexus-debugging` | 追踪错误和异常行为 |
| `gitnexus-refactoring` | 安全地重命名、提取、拆分或移动代码 |
| `gitnexus-pr-review` | 基于图谱感知的 PR 审查 |
| `gitnexus-cli` | 运行 GitNexus CLI 命令 |
| `gitnexus-guide` | 工具、资源和图谱模式参考 |

### `firecrawl-*` — Web 自动化

31 个来自 Firecrawl CLI、build、workflows 仓库的技能。覆盖 scrape、crawl、map、search、parse、interact、monitor、deep-research、lead-gen、market-research、SEO audit、knowledge-base、QA 等。来源：[`firecrawl/cli`](https://github.com/firecrawl/cli)、[`firecrawl/skills`](https://github.com/firecrawl/skills)、[`firecrawl/firecrawl-workflows`](https://github.com/firecrawl/firecrawl-workflows)。

### `context7-cli` / `context7-mcp` / `find-docs` — 文档查询

通过 Context7 获取任意库、框架、SDK、CLI 或云服务的最新文档、API 参考和代码示例。来源：[`upstash/context7`](https://github.com/upstash/context7)。

### `react-doctor` 家族 — React triage

12 个来自 react-doctor 仓库的技能。主 `react-doctor` 技能运行 `npx react-doctor@latest` 给 React 代码库打 0–100 分；同仓的 `deslop`、`find-similar-functions`、`fuzz`、`improve-react`、`product-thinking`、`rde-eval`、`rule-research`、`rule-validate`、`rule-writing`、`ship`、`writing-guidelines` 覆盖相关的规则创作和质量工作流。来源：[`aidenybai/react-doctor`](https://github.com/aidenybai/react-doctor)。

```bash
# 对变更文件运行（回归检查）
npx react-doctor@latest --verbose --scope changed

# 运行完整扫描
npx react-doctor@latest --verbose
```

项目有 `🩺-react-doctor.yml` CI 工作流，在 PR 上运行。提交前本地评分不应回退。

### `ponytail` 家族 — Lazy / 最小方案

6 个技能（`ponytail`、`ponytail-audit`、`ponytail-debt`、`ponytail-gain`、`ponytail-help`、`ponytail-review`），强制采用真正可行的最懒方案——最简单、最短、最最小。来源：[`dietrichgebert/ponytail`](https://github.com/dietrichgebert/ponytail)。

### `tdd` — 测试驱动开发

垂直切片的红-绿-重构循环。来源：[`mattpocock/skills`](https://github.com/mattpocock/skills)。

<Callout title="TDD 反模式">

不要先写完所有测试再实现（水平切片）。应使用**垂直切片**：写一个失败测试、实现通过、然后重复。这样产生的测试验证的是实际行为，而非想象中的行为。

</Callout>

### `grill-with-docs` — 计划 + 文档产物

计划到提交链的推荐入口。对照代码库 grill 用户的计划，并产出 `CONTEXT.md` 和 ADRs 作为后续阶段引用的持久产物。来源：[`mattpocock/skills`](https://github.com/mattpocock/skills)。

### `domain-modeling` — 磨砺领域语言

可选阶段 1b。磨砺通用语言和领域术语，让后续 spec 和 tickets 引用共享词汇。来源：[`mattpocock/skills`](https://github.com/mattpocock/skills)。

### `to-spec` — 计划 → spec

将经过 grill 的计划或对话综合成结构化 spec，便于拆分 tickets。来源：[`mattpocock/skills`](https://github.com/mattpocock/skills)。

### `to-tickets` — spec → tracer-bullet tickets

将 spec 拆成带显式阻塞边的垂直切片 tickets，可直接进入 TDD 实现。来源：[`mattpocock/skills`](https://github.com/mattpocock/skills)。

### `grilling` / `grill-me` — 压力测试计划

`grilling` 围绕计划的每个方面对用户进行无情访谈，直到达成共识。`grill-me` 是触发 `/grilling` 的 stub 别名。当不需要文档产物时，用作 `grill-with-docs` + `domain-modeling` 的轻量替代。来源：[`mattpocock/skills`](https://github.com/mattpocock/skills)。

## 平台技能（仅路由）

这些技能在运行时从 Codex 平台加载，没有本地 `SKILL.md` 文件。

| 技能 | 用途 |
| --- | --- |
| GitHub | 仓库、议题、PR、审查评论、CI 和发布工作流 |
| Browser / Playwright / Chrome | 应用内浏览器检查、终端自动化、Chrome 状态 |
| Vercel | Next.js、React、shadcn/ui、AI SDK、部署、存储 |
| Cloudflare | Workers、Wrangler、Durable Objects、Agents SDK |

| imagegen | 栅格图像生成或编辑 |
| skill-creator / skill-installer / plugin-creator | 技能和插件创作 |

## 技能加载机制

1. 每个 `SKILL.md` frontmatter 中的 `description` 字段列在代理的可用技能中。
2. 当用户任务匹配技能触发条件时，代理加载完整的 `SKILL.md`。
3. `SKILL.md` 路由到 `references/` 中的具体参考文件，获取详细工作流。
4. 参考文件包含实际的分步指令。

只有 `AGENTS.md`（Claude Code 为 `CLAUDE.md`）和 `.trae/rules/git-commit-message.md` 会注入每次对话。其余内容按需加载。

## 添加新技能

优先用 `skills` CLI 而非手写技能目录：

```bash
# 搜索技能
npx -y skills find <query>

# 在项目级安装（不加 -g）；限定 agent 避免产生杂散目录
npx -y skills add <owner/repo> --full-depth --skill '*' --agent codex,claude-code,trae-cn --yes

# 新克隆后恢复
npx -y skills experimental_install
```

如确需手写技能，目录必须包含：

- `SKILL.md` — 含 `name` 和 `description` 的 frontmatter，加上路由指令
- `references/` — 可选的按需加载详细指南

任何技能集变更后，更新 `AGENTS.md` / `CLAUDE.md` / `.github/note/AGENTS-zh.md` 中的技能表，并提交 `skills-lock.json`。
