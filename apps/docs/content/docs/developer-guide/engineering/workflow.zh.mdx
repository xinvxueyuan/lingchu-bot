## 工作流

保持开发变更小而聚焦、验证明确、影响可理解。仓库使用 `Taskfile.yml` 作为主要自动化入口：Python/代码工作通过 `uv` 运行，文档/前端工作通过 `pnpm` 运行，跨工作区任务由 Turbo 编排。

## 开发环境搭建

本节引导外部开发者从零开始搭建可用的开发环境。

### 前置条件

- Python 3.13（`requires-python = ">=3.13, <3.14"`）
- [uv](https://docs.astral.sh/uv/) — Python 包和项目管理器
- [Node.js](https://nodejs.org/) 22+ 和 [pnpm](https://pnpm.io/) — 用于文档站点和前端工作区
- [git](https://git-scm.com/)

### 安装

1. 克隆仓库：

   ```bash
   git clone https://github.com/xinvxueyuan/lingchu-bot.git
   cd lingchu-bot
   ```

2. 安装所有依赖（Python + Node.js）：

   ```bash
   task install
   ```

   这会运行 `uv sync` 和 `pnpm install`。如需分别安装：

   ```bash
   uv sync --frozen       # Python 依赖
   pnpm install --frozen-lockfile  # Node.js 依赖
   ```

3. 复制环境模板：

   ```bash
   cp .env.example .env
   ```

   编辑 `.env`，设置 `LINGCHU_SUPERUSERS`、`SUPERUSERS` 以及适配器连接详情以适配你的环境。

4. 验证搭建：

   ```bash
   task check    # 静态检查、格式化、Markdown lint、类型检查
   task test     # Python 测试 + 文档测试
   ```

   如果两者均通过，则环境已准备好进行开发。

### 开发工作流

典型的开发周期：

1. 创建分支：`git checkout -b feature/your-feature`
2. 遵循现有模式和风格进行修改。
3. 对修改过的文件运行聚焦检查：

   ```bash
   uv run -m ruff check path/to/file.py
   uv run -m ruff format path/to/file.py
   ```

4. 提交前运行完整检查套件：

   ```bash
   task check
   task test
   ```

5. 使用 gitmoji + Conventional Commits 格式提交（由 `.husky/commit-msg` 强制执行）。
6. 推送并按照 [PR 描述指南](#pr-描述) 开启 PR。

### 测试

| 测试类型 | 命令 | 范围 |
|-----------|---------|-------|
| Python 单元测试 | `uv run -m pytest` | 所有 Python 测试 |
| Python 聚焦测试 | `uv run -m pytest tests/handle/commands/test_mute.py` | 单个测试文件 |
| 文档测试 | `pnpm --filter docs test` | Vitest 组件/单元测试 |
| 文档链接检查 | `pnpm --filter docs run lint:links` | 内部链接验证 |
| E2E 测试 | `pnpm --filter docs exec playwright test` | Playwright 浏览器测试 |
| 完整 CI 序列 | `task ci` | check + test + build |

### 构建

```bash
task build      # 构建所有工作区（Python + 文档）
pnpm turbo run build --filter=docs  # 仅构建文档站点
uv build --clear  # 仅构建 Python 包
```

### i18n 工作流

当修改可翻译字符串时：

```bash
task i18n   # 提取、更新和编译 gettext 目录
```

这会运行 `pybabel extract`、`pybabel update` 和 `pybabel compile`。详见[国际化](i18n)。

## 推荐步骤

<Steps>
<Step>

### 明确目标

在编写任何代码之前，明确目标、成功标准和范围外事项。

</Step>
<Step>

### 检查现有变更

运行 `git status --short` 确认现有变更。默认假设现有变更来自其他贡献者或更早的工作会话。

</Step>
<Step>

### 阅读相关代码

阅读相关代码和测试，优先遵循现有结构。使用 GitNexus 或 codegraph 理解代码结构和关系。

</Step>
<Step>

### 分析影响

在修改函数、类或方法之前，运行 GitNexus 或 codegraph 影响分析。

</Step>
<Step>

### 实现并测试

实现最小有用的变更并添加测试。

</Step>
<Step>

### 运行检查

通过 `task check`、`task test` 或聚焦的 `uv` / `pnpm` 命令运行相关检查。

</Step>
<Step>

### 验证变更范围

提交前运行 GitNexus detect changes。

</Step>
</Steps>

<Callout title="可翻译字符串">

当修改可翻译字符串时，也要同步 Babel 目录。参见[国际化](i18n)。

</Callout>

## 自动化入口

优先使用以下 Taskfile 任务：

```bash
task install
task check
task test
task build
task ci
task i18n
```

对于仅涉及 Python 的变更，适当时使用聚焦的 `uv run ...` 检查。对于文档或前端包的变更，使用聚焦的 `pnpm --filter docs ...` 或 `pnpm turbo run ...` 检查。

## 保护工作区

不要还原、格式化或重写与当前任务无关的文件。当存在现有变更时，默认假设它们来自其他贡献者或更早的工作会话。

## 分支与提交

常用分支前缀：

| 前缀 | 用途 |
| --- | --- |
| `feature/` | 新功能 |
| `fix/` | 缺陷修复 |
| `hotfix/` | 紧急生产修复 |
| `releases/` | 发布分支 |
| `docs/` | 文档变更 |
| `refactor/` | 代码重构 |

GitHub Actions 也会匹配 `feature/**`、`fix/**`、`hotfix/*`、`releases/**`、`main` 和 `dev` 等分支。实际目标分支应遵循维护者或 PR 页面的指引。

提交信息必须遵循 gitmoji + Conventional Commits。参见[提交风格](commit-style)。

```text
🐛 fix: 修复成员禁言成功反馈
📝 docs: 重写文档站
✅ test: 增加群管理异常覆盖
```

## PR 描述

PR 应说明：

- 变更目的。
- 关键实现细节。
- GitNexus 影响分析结果。
- 已运行的检查。
- 未决事项或已知风险。

如果 PR 涉及文档站点或国际化，请注明是否运行了 Markdown lint、docs lint/test/build 和 `task i18n`。
