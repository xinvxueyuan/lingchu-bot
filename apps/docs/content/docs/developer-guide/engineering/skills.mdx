## Agent Skills

The project ships a set of **skills** — structured prompts loaded on demand by AI coding agents (Trae, Codex, Claude Code). Skills turn common workflows into repeatable, disciplined processes.

Skills are **not** injected into every conversation. They are loaded only when the user's task matches the skill trigger. This keeps context lean while making specialized workflows available when needed.

## Development workflow chain

The plan-to-commit skills form a five-phase pipeline. Load each skill when the corresponding phase starts; do not preload the entire chain.

```text
grill-with-docs          ← phase 1: PLAN
  ↓                        grill the plan, build CONTEXT.md + ADRs
domain-modeling          ← phase 1b: sharpen domain language (optional)
  ↓
to-spec                  ← phase 2: SPEC
  ↓                        synthesize plan into a spec
to-tickets               ← phase 3: TICKETS
  ↓                        break spec into vertical-slice tickets
tdd                      ← phase 4: IMPLEMENT
  ↓                        red-green-refactor, one slice at a time
  ├─ ponytail             ← enforce minimal solution during implementation
  ├─ context7-cli         ← look up library docs when needed
  ├─ gitnexus             ← run impact() before editing any symbol
  ├─ firecrawl-*          ← web research/scraping when needed
  └─ react-doctor         ← React triage for frontend changes
prek                     ← phase 5: COMMIT
                           Git hooks: lint + format + type + test
```

Lighter alternative: `grilling` / `grill-me` replace `grill-with-docs` + `domain-modeling` when you only need a pressure-test without docs artifacts.

## Skill management

Skills are managed by the [`skills` CLI](https://github.com/vercel-labs/skills) and tracked at the project level so every contributor gets the same set.

| Concern | Location |
| --- | --- |
| Real skill files (single source of truth) | `.agents/skills/<skill-name>/SKILL.md` |
| Claude Code entry (whole-directory symlink) | `.claude/skills` → `../.agents/skills` |
| Trae entry (whole-directory symlink) | `.trae/skills` → `../.agents/skills` |
| Lock file (restore with `skills experimental_install`) | `skills-lock.json` |
| GitNexus-managed skills (gitignored, regenerated by CLI) | `.agents/skills/gitnexus/` |

To add a skill: `npx -y skills add <owner/repo> --full-depth --skill '*' --agent codex,claude-code,trae-cn --yes` (without `-g` so it lands in the project, not the user home). To restore on a fresh clone: `npx -y skills experimental_install`.

<Callout title="Avoid --all">

`--all` is shorthand for `--skill '*' --agent '*' -y` and installs to **every** agent the CLI knows (~30+), including `eve` (`agent/skills/`) and `astrbot` (`data/skills/`) which create unwanted real-copy directories. Always scope with `--agent codex,claude-code,trae-cn` to target only the agents this project uses.

</Callout>

## Project-local skills

### `prek` — Git hooks

Configure, run, or troubleshoot `prek`, Husky, pre-commit, commit-msg, and lint/format/type/test hooks. Source: [`j178/prek`](https://github.com/j178/prek).

### GitNexus — Code intelligence

GitNexus builds a knowledge graph from the repo and exposes MCP tools for impact analysis, execution-flow exploration, safe renames, and PR review. The skill folder is regenerated by `gitnexus analyze` and is gitignored. See [GitNexus Workflow](gitnexus) for the full guide.

| Skill | Purpose |
| --- | --- |
| `gitnexus-exploring` | Understand architecture and execution flows |
| `gitnexus-impact-analysis` | Assess blast radius before changing symbols |
| `gitnexus-debugging` | Trace errors and unexpected behavior |
| `gitnexus-refactoring` | Rename, extract, split, or move code safely |
| `gitnexus-pr-review` | Review pull requests with graph-aware analysis |
| `gitnexus-cli` | Run GitNexus CLI commands |
| `gitnexus-guide` | Reference for tools, resources, and schema |

### `firecrawl-*` — Web automation

31 skills from the Firecrawl CLI, build, and workflows repos. Covers scrape, crawl, map, search, parse, interact, monitor, deep-research, lead-gen, market-research, SEO audit, knowledge-base, QA, and more. Sources: [`firecrawl/cli`](https://github.com/firecrawl/cli), [`firecrawl/skills`](https://github.com/firecrawl/skills), [`firecrawl/firecrawl-workflows`](https://github.com/firecrawl/firecrawl-workflows).

### `context7-cli` / `context7-mcp` / `find-docs` — Documentation lookup

Retrieve up-to-date docs, API references, and code examples for any library, framework, SDK, CLI, or cloud service via Context7. Source: [`upstash/context7`](https://github.com/upstash/context7).

### `react-doctor` family — React triage

12 skills from the react-doctor repo. The main `react-doctor` skill runs `npx react-doctor@latest` to score a React codebase 0–100; the siblings (`deslop`, `find-similar-functions`, `fuzz`, `improve-react`, `product-thinking`, `rde-eval`, `rule-research`, `rule-validate`, `rule-writing`, `ship`, `writing-guidelines`) cover related rule-authoring and quality workflows. Source: [`aidenybai/react-doctor`](https://github.com/aidenybai/react-doctor).

```bash
# Run on changed files (regression check)
npx react-doctor@latest --verbose --scope changed

# Run a full scan
npx react-doctor@latest --verbose
```

The project has a `🩺-react-doctor.yml` CI workflow that runs on PRs. Local scores should not regress before committing.

### `ponytail` family — Lazy / minimal solutions

6 skills (`ponytail`, `ponytail-audit`, `ponytail-debt`, `ponytail-gain`, `ponytail-help`, `ponytail-review`) that force the laziest solution that actually works — simplest, shortest, most minimal. Source: [`dietrichgebert/ponytail`](https://github.com/dietrichgebert/ponytail).

### `tdd` — Test-driven development

Red-green-refactor cycles with vertical slices. Source: [`mattpocock/skills`](https://github.com/mattpocock/skills).

<Callout title="TDD anti-pattern">

Do not write all tests first and then all implementation (horizontal slicing). Instead, use **vertical slices**: write one failing test, implement to pass, then repeat. This produces tests that verify actual behavior, not imagined behavior.

</Callout>

### `grill-with-docs` — Plan + docs artifacts

The recommended entry point for the plan-to-commit chain. Grills the user's plan against the codebase and produces `CONTEXT.md` plus ADRs as durable artifacts that later phases reference. Source: [`mattpocock/skills`](https://github.com/mattpocock/skills).

### `domain-modeling` — Sharpen domain language

Optional phase 1b. Sharpens ubiquitous language and domain terms so later specs and tickets reference a shared vocabulary. Source: [`mattpocock/skills`](https://github.com/mattpocock/skills).

### `to-spec` — Plan → spec

Synthesizes a grilled plan or conversation into a structured spec ready for ticket breakdown. Source: [`mattpocock/skills`](https://github.com/mattpocock/skills).

### `to-tickets` — Spec → tracer-bullet tickets

Breaks a spec into vertical-slice tickets with explicit blocking edges, ready for TDD implementation. Source: [`mattpocock/skills`](https://github.com/mattpocock/skills).

### `grilling` / `grill-me` — Stress-test a plan

`grilling` relentlessly interviews the user about every aspect of a plan, decision, or idea until a shared understanding is reached. `grill-me` is a stub alias that triggers `/grilling`. Use this as the lighter alternative to `grill-with-docs` + `domain-modeling` when no docs artifacts are needed. Source: [`mattpocock/skills`](https://github.com/mattpocock/skills).

## How skills are loaded

1. The `description` field from each `SKILL.md` frontmatter is listed in the agent's available skills.
2. When the user's task matches a skill trigger, the agent loads the full `SKILL.md`.
3. The `SKILL.md` routes to specific reference files in `references/` for detailed workflows.
4. Reference files contain the actual step-by-step instructions.

Only `AGENTS.md` (or `CLAUDE.md` for Claude Code) and `.trae/rules/git-commit-message.md` are injected into every conversation. Everything else loads on demand.

## Adding a new skill

Prefer the `skills` CLI over hand-authoring a new skill folder:

```bash
# Search for a skill
npx -y skills find <query>

# Install at project level (no -g flag); scope agents to avoid stray dirs
npx -y skills add <owner/repo> --full-depth --skill '*' --agent codex,claude-code,trae-cn --yes

# Restore on a fresh clone
npx -y skills experimental_install
```

If a hand-authored skill is unavoidable, the folder must contain:

- `SKILL.md` — frontmatter with `name` and `description`, plus routing instructions
- `references/` — optional detailed guides loaded on demand

After any change to the skill set, update the skill tables in `AGENTS.md` / `CLAUDE.md` / `.github/note/AGENTS-zh.md` and commit `skills-lock.json`.
